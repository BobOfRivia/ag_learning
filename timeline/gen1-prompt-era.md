# 第一代：Prompt Era（2020–2022）

## 定义性变化

第一代的标志是**LLM 作为一种新的计算基元出现，但几乎不会"做事"**。模型只有一个能力：给定文本，生成下一段文本。所有更复杂的行为——推理、规划、使用工具、记忆——都靠工程层在 prompt 中搭脚手架模拟。这个时代的核心张力是：模型的原始能力极其有限，但研究者发现通过巧妙的 prompt 设计可以诱导出远超预期的行为。

## 地基能力水位

这个时代的 LLM 地基有几个硬约束，理解这些约束才能理解为什么第一代的工程形态长成那样：

**Reasoning：零。** 模型没有任何内化的推理能力。GPT-3（2020）在需要多步推理的任务上表现极差——除非你在 prompt 中显式展示推理步骤。这直接催生了整个 prompt engineering 领域：既然模型不会想，那就教它怎么想。

**Tool Use：零。** 没有原生的函数调用能力。模型不知道"工具"是什么。要让模型调用工具，必须在 prompt 中定义一种文本格式（如 `Action: search[query]`），然后用正则表达式从模型输出中解析出调用意图，手动执行后把结果拼回 prompt。整个过程脆弱、不可靠、充满解析错误。

**Context Window：极小。** GPT-3 的上下文窗口是 4K tokens（约 3000 个英文单词）。这意味着 agent 能"记住"的信息量极其有限——几轮对话就可能撑满窗口。没有空间放长文档、没有空间放工具结果历史、没有空间放详细的推理链。

**Memory：无。** 模型是完全无状态的。每次请求都是一次独立的函数调用，不保留任何历史信息。所谓"记忆"就是把前几轮对话拼进 prompt，受限于 context window 的大小。

**Instruction Following：弱。** 早期模型对指令的遵循能力很差。InstructGPT（2022-01）和 ChatGPT（2022-11）通过 RLHF 大幅改善了这一点，但仍然远不够精确——模型经常忽略复杂指令中的具体约束。

## 工程重心与格局

工程量几乎全部集中在 **L1（Agent 内核）**。原因很简单：地基能力太弱，工程层必须用 prompt 技巧补偿模型的每一个短板。L2（工具接口）只有最原始的形态（手工解析文本格式的工具调用），L3（多 Agent）和 L4（生产化）几乎不存在——因为单个 agent 的可靠性都没解决，谈不上协作和上线。

这个时代的工程实践有一个共同特征：**所有能力都是外挂的。** 模型本身不变，所有改进都通过修改 prompt 实现。这使得 prompt engineering 成为这个时代的核心技能——它既是科学（可复现的技术），也是手艺（依赖经验和直觉），也是迷信（很多"有效"的技巧缺乏可靠的解释）。

## 关键事件与里程碑

**2020-06：GPT-3 发布。** 175B 参数，展示了 few-shot learning 的能力——不需要微调，只需在 prompt 中给几个示例。这是 LLM 作为通用计算基元的起点。

**2022-01：Chain-of-Thought（CoT）论文发表（Wei et al.）。** 发现在 prompt 中加入推理步骤的示例，可以让模型"学会"逐步推理。这是 prompt engineering 最重要的单一发现，直接催生了后续的 Self-Consistency、ToT 等一系列技术。

**2022-01：InstructGPT 发布。** 通过 RLHF 让模型更好地遵循人类指令。这是 instruction following 维度的第一次重大改进。

**2022-10：ReAct 论文发表（Yao et al.）。** 将 reasoning 和 acting 交错进行——想一步、做一步、观察结果、再想。这是"agent"这个概念在 LLM 语境下的首次清晰定义。尽管此时的"行动"只是文本格式的模拟调用，ReAct 建立了后续所有 agent 架构的基本范式。

**2022-11：ChatGPT 发布。** 对话式界面 + RLHF 调优。不是技术上的根本突破，但把 LLM 从研究工具变成了大众产品，催生了整个应用生态。

**2022-12：Self-Consistency 论文（Wang et al.）。** 同一问题多次采样，对最终答案投票取多数。用推理路径的多样性换准确率。

## 代表性技术

这些技术构成了第一代的核心工具箱。它们的共同特征是：**模型本身不会做这件事，工程层通过 prompt 模拟出这个能力。**

**Chain-of-Thought（CoT）。** 通过在 prompt 中展示推理步骤，诱导模型逐步推理。Zero-shot CoT（"Let's think step by step"）和 few-shot CoT（给推理示例）是两种主要形态。在这个时代，CoT 是让模型做任何需要推理的任务的必要前提。

**ReAct。** Thought → Action → Observation 紧密交错的循环。每轮思考只规划一步行动，思考和行动的粒度严格对齐。这是 agent 内核的原型模式。

**Self-Consistency。** 多次采样投票。本质上是用计算量换准确率——模型单次推理不可靠，那就多推理几次取共识。

**Tree-of-Thoughts（ToT）。** 将推理建模为搜索树，每个节点是一个思考步骤，可以回溯和分支。工程复杂度高，实际生产部署极少，更多停留在学术实验。

**Reflexion / Self-Refine。** 让模型审视自己的输出，生成反思，然后重试。外部反思循环——用另一轮 LLM 调用修复上一轮的错误。

**Prompt Chaining。** 将复杂任务分解为串行步骤，每步的输出作为下一步的输入。这是最原始的编排模式。

## 塌缩与涌现

### 塌缩（从前 LLM 时代继承的模式消失）

- **传统 NLP pipeline**（分词 → NER → 关系抽取 → 推理）被端到端的 LLM 调用替代。不再需要为每个子任务训练专门的小模型。
- **规则引擎驱动的对话系统**（状态机 + 意图分类 + 槽填充）开始被 LLM 直接处理对话替代，至少在原型和 demo 层面。

### 涌现

- **Prompt Engineering** 作为一个工程学科出现。从"怎么写 prompt"发展出系统化的方法论。
- **Agent 概念的雏形**。ReAct 定义了 agent 的基本范式——思考-行动循环。尽管实现粗糙，但概念框架从此确立。
- **外挂式 reasoning 技术族**。CoT、Self-Consistency、ToT、Reflexion 构成一个完整的技术谱系，核心思路都是"用 prompt 模拟模型不具备的推理能力"。
- **LLM 应用的基本架构模式**。Prompt template + LLM call + output parsing 成为所有 LLM 应用的基本骨架。

## 遗产与局限

第一代留下的最重要遗产是**概念框架**而非具体技术。ReAct 定义的思考-行动循环至今仍是 agent 的核心范式（尽管实现方式已被 reasoning model 彻底改变）。CoT 揭示的"让模型展示中间步骤可以提升最终结果"这一洞察，直接启发了后续 reasoning model 的训练方向。

但这个时代的具体技术在第三代几乎全面塌缩。CoT prompting、Self-Consistency、ToT、Reflexion——这些曾经是 agent 工程的核心技能，在 reasoning model 出现后大幅贬值。这说明第一代本质上是一个**过渡期**：它用外挂技巧证明了某些能力的价值（"模型如果会推理，agent 就能做很多事"），然后模型厂商把这些能力内化到了地基中。

局限同样明显：agent 极不可靠（工具调用靠文本解析，一个格式错误就全崩）；context window 太小导致稍微复杂的任务就超出处理能力；没有生产化的任何基础设施（eval、tracing、cost management 都不存在）。

## 与相邻代际的关系

**→ 第二代的过渡信号**：ChatGPT 的爆发（2022-11）和 OpenAI function calling API 的推出（2023-06）是分水岭。function calling 把第一代最痛的问题——手工解析文本格式的工具调用——用 API 原生支持一次性解决。同时 context window 从 4K 跳到 32K 再到 100K+，解除了第一代最大的硬约束之一。

**对第三代的长程影响**：第一代证明了 reasoning 的价值（通过 CoT 等外挂技巧），第三代将其内化。可以说第一代的 prompt engineering 研究是 reasoning model 训练方向的"概念验证"——先在 prompt 层证明"让模型想"有用，再通过 RLVR 等训练手段让模型自己学会想。
