# 体系总览

> 这份文档是整个知识体系的入口。先读完这里建立框架，再沿阅读路径进入各章深入。

## 一个根本张力

Agent 工程的所有复杂性，归结为一件事：**在不可靠的 LLM 上构建可靠的系统。**

这个体系用一个隐喻来组织这个领域的全部知识——地基与建筑。

```
                  Agent 工程（建筑）
        ┌──────────────────────────────────┐
   L4   │  生产化                            │  Evals, Tracing, Cost, Safety
   L3   │  多 Agent 协作                     │  Orchestration, Communication
   L2   │  与世界的接口                      │  Tools, MCP, RAG, Memory
   L1   │  Agent 内核                        │  Reasoning loop, Context engineering
        ├──────────────────────────────────┤
   L0   │  LLM 本身（地基）                  │  七大维度 + 准维度
        └──────────────────────────────────┘

         地基变化 ─────────► 建筑塌缩 / 涌现
```

LLM 本身的能力是地基（L0），在其上搭建的四层工程体系是建筑（L1–L4）。但这不是一张静态分层图。它的核心在于**动态传导**：地基每发生一次能力质变，都会向上冲击建筑——某些工程模式因此失去存在的必要（塌缩），某些新的工程问题随之浮现（涌现）。

这种塌缩-涌现循环是理解 Agent 工程演进的核心机制。过去六年里，它已经发生了至少三轮。

## 三轮冲击，三代架构

**第一代（2020–2022）：Prompt 时代。** 模型本身几乎不会"做事"。所有智能行为都靠 prompt 层搭脚手架模拟——Chain-of-Thought 教模型思考，ReAct 教模型交替推理和行动。工程量几乎全部集中在 L1（Agent 内核），靠外挂技巧弥补地基的不足。

**第二代（2023–2024 中）：Tool-RAG 时代。** 地基发生第一次质变：模型获得原生的工具调用能力，上下文窗口扩大到 100K+ tokens。冲击波传导到 L2——RAG 和 Tool 工程化成为这一代的核心主题。Agent 从"只能思考"变成"能操作世界"，但推理能力仍然靠外挂。

**第三代（2024 末–2026）：Reasoning 时代。** 地基的第二次质变：reasoning 从 prompt 技巧转为训练内化（标志性事件：OpenAI o1）。冲击波剧烈——L1 层大面积塌缩（CoT prompting、Self-Consistency、Reflexion 等一批第一代核心技术几乎一夜贬值），L4 层大量涌现（成本飙升 20–50 倍，需要全新的可观测性和评估体系）。这是迄今为止塌缩-涌现最剧烈的一次转代。

**第四代（2026– ？）：Memory 时代（猜测）。** 如果 memory 从工程外挂向地基内化，L2 的向量库和 RAG 工程将大面积塌缩，L3（多 Agent 协作）的设计动机将被根本重塑——有持久记忆的 agent 之间的协作，与无状态 agent 之间的协作，是完全不同的问题。这一代尚未被事实验证，详见 [timeline/gen4-memory-era.md](timeline/gen4-memory-era.md) 中的判定标准和失效条件。

理解了这个动态，你就有了阅读后续章节的核心视角：每个工程层的当前形态，都是地基变化冲击后的产物；每个维度的演进方向，都在预告下一波冲击从哪来。

## 阅读路径

先看建筑（Agent 工程在解决什么问题），再看地基（为什么建筑长成这样）。这个顺序的逻辑是：先建立工程全景，再往下挖驱动力。

### 第一部分：建筑 — Agent 工程的四个层次

由内向外，每一层解决一类不同的问题。

**[L1：Agent 内核](layers/L1-agent-core.md)** — 如何让模型从"回答问题"变成"完成任务"

这是离 LLM 最近的一层。它解决的核心问题是：给模型一个目标，让它自主规划步骤、执行动作、处理失败。这一层在第三代经历了最剧烈的塌缩——大量曾经的核心技术（CoT、ReAct、Self-Consistency）被 reasoning model 内化。读完这章，你应该理解当前 agent 内核的"思考-行动"循环是怎么工作的，以及哪些曾经重要的概念已经不再需要工程层操心。

**[L2：与世界的接口](layers/L2-world-interface.md)** — Agent 如何感知和操作真实世界

L1 决定 agent 怎么思考，L2 决定它能接触什么。这一层包含四个演进速度差异很大的子领域：Tool Use / MCP（标准化爆发期）、RAG（大面积塌缩中）、Memory（可能是下一代核心主题）、以及更广的外部接口。读完这章，你应该对 agent 的"手和眼"有完整认知——各子领域各自走到了哪，哪些在退潮，哪些在上升。

**[L3：多 Agent 协作](layers/L3-multi-agent.md)** — 何时需要多个 Agent，以及如何编排

这一层的设计动机正在经历深刻转变。第二代的多 agent 主要为了提升质量（多个 agent 模拟团队讨论），第三代的多 agent 更多为了效率和隔离性（reasoning model 太慢太贵，需要并行化和成本混合）。读完这章，你应该能判断一个任务是否真的需要多 agent，以及当前主流的编排模式各自适用于什么场景。

**[L4：生产化](layers/L4-production.md)** — 如何让 Agent 在真实环境中可靠运行

L1–L3 关心的是"能不能做到"，L4 关心的是"能不能上线"。这是第三代重要性提升最快的一层：reasoning model 带来的成本飙升、延迟不确定性、思考链的可观测性、新的安全面（模型在思考链中"想偏"绕过 guardrail）——都是第三代涌现出的新工程问题。读完这章，你应该了解 agent 系统上生产需要关注哪些维度，以及当前各维度的工程成熟度。

### 第二部分：地基 — LLM 能力的七个维度

读完建筑四层，你已经知道 agent 工程在做什么。这一部分回答"为什么"——是什么底层能力在驱动建筑的形态变化。

我们把 LLM 的能力分解为七个维度。以下按叙事逻辑排列，不是按文件编号顺序。

**[Reasoning](dimensions/01-reasoning.md)** — 模型"想多深"的能力，过去两年 agent 架构变化的最大驱动力

从这个维度开始，因为它引发了第三代转型——迄今最剧烈的一次塌缩-涌现。理解 reasoning 如何从 prompt 外挂变成训练内化，是理解当前整个 agent 工程格局的前提。

**[Inference-Time Compute](dimensions/07-inference-time-compute.md)** — Reasoning 的代价：推理时花多少算力，由谁来控制

Reasoning 和 Inference-Time Compute 是一枚硬币的两面。前者是"能力"，后者是"成本与可控性"。读完 Reasoning 再读这个维度，你会理解为什么 L4 层在第三代突然变得如此重要。

**[Tool Use](dimensions/04-tool-use.md)** — 模型调用外部工具的原生能力

这是 agent 从"能思考"到"能做事"的关键跃迁，直接塑造了 L2 的工程形态，也是第二代转型的核心驱动力。

**[Long Context](dimensions/02-long-context.md)** — 单次调用能感知多少信息

上下文窗口从 4K 到 1M+ 的扩展催生了大量工程变化——最直接的是让许多 RAG 场景塌缩。但"能放进去"和"能用好"是两回事，当前瓶颈已从容量转向利用率。

**[Memory](dimensions/06-memory.md)** — 跨 session 的持久记忆能力

Memory 经常和 Long Context 混淆，但本质不同：Long Context 是单次感知范围，Memory 是跨次积累。这个维度是第四代的核心假设所在——如果 memory 内化，会引发下一波重大塌缩-涌现。紧接 Long Context 读，有助于建立两者的清晰区分。

**[Multimodality](dimensions/03-multimodality.md)** — 跨模态的原生理解与生成

当模型不只处理文本，还能原生理解图像、音频、视频甚至操作 GUI 时，agent 的"接口"（L2）会发生质变。这个维度与 Tool Use 的交叉催生了 GUI agent / browser agent 等新范式。

**[Instruction Following](dimensions/05-instruction-following.md)** — 模型遵循指令的精度和可控性

这是一个贯穿所有层级的横切维度。它不像 Reasoning 或 Tool Use 那样有明确的工程层对应，但它的水位高低影响每一层的工程设计——从 prompt 的写法到 agent 行为的可预测性。放在最后读，因为此时你已有足够上下文来理解它的横切影响。

### 快速路径

如果时间有限，以下四份内容足以建立基本框架：

1. **本文序章**（"一个根本张力"到"三轮冲击"两节）— 理解框架和动态机制

## 导航与交叉阅读

### 维度之间的关键耦合

各维度不是独立演进的。以下耦合关系在多个章节中反复出现，值得交叉阅读：

- **Reasoning × Inference-Time Compute**：本质上是同一件事的能力面和成本面
- **Reasoning × Memory**：组合催生"能反思过去并改进"的 agent，是第四代的核心机制
- **Multimodality × Tool Use**：交叉催生 GUI agent / browser agent 新范式
- **Long Context × Memory**：经常被混淆，但一个是单次感知范围，一个是跨次积累

### 塌缩-涌现日志

[logs/collapse-emergence-log.md](logs/collapse-emergence-log.md) 是这个体系最重要的活体文件。它记录每一次地基变化或生态事件对工程层的冲击。当你想了解"最近发生了什么变化"，从这个日志切入。

### 决策模式

当你面对具体的工程决策（选型、架构设计、评估方案），可以通过 Claude Code 直接提问——它会把整个项目作为知识库，跨文件检索综合后给出建议，不需要自己手动跨文件查找。

### 归位规则

当一个概念同时跨 L0（地基能力）和工程层时：

- **维度文件（dimensions/）记录能力本身的演进**：模型原生能做什么、做到什么程度、趋势是什么
- **层级文件（layers/）记录工程应对方式的演进**：工程上怎么用、怎么补、怎么绕
- 同一个事件可以同时触发两边的更新。在塌缩-涌现日志中，用"关联文件"字段标注同时更新了哪些文件

### 准维度

除七大维度外，以下能力边界模糊但值得追踪：

- **Agentic Capability**：模型原生的 agent 行为能力（主动规划、错误恢复等）
- **Self-Evolution**：模型自主改进自身的能力
- **Speed / Cost**：推理速度和成本的变化
- **Ecosystem Maturity**：部分工程变化不由 LLM 能力驱动，而由生态标准化驱动（如 MCP 协议的出现）。此类事件在塌缩-涌现日志中标注为"生态事件"

### 元原则

1. **先全景，再追踪**：先建立完整的知识地图，再在此基础上追踪变化
2. **空白是待填的**：覆盖缺失应被识别和补上，但完整指结构性覆盖，不是细节穷举
3. **双索引同步**：每个知识点同时回答两个问题——空间锚点（哪一层、哪个维度）和时间锚点（哪一代、塌缩还是涌现）
4. **结构稳定，内容生长**：维度框架可以稳定较长时间，文件内容随学习逐步丰满
