# 记忆安全（Memory Security）

> 锚点：层级 L4（生产化）/ 维度 6 Memory / 与 topics/memory-architecture.md 有直接关联（攻击面取决于架构选择）

## 这个概念是什么

Agent 记忆构成一个**独立的安全类别**，区别于传统的 prompt injection。三个特有属性定义了它的独特性：**持久性**（毒素跨 session 存活，不像 prompt injection 只影响当次交互）、**状态性**（累积的偏差导致渐进式行为漂移，而非一次性劫持）、**传播性**（污染通过多 agent 共享记忆扩散至其他 agent 和用户，形成蠕虫式传播）。

独立展开的理由：记忆安全的威胁分类和治理框架有丰富的内部结构，维度文件无法充分覆盖。且记忆安全很可能是 agent 记忆从 demo 到企业生产的最大阻碍——不理解这个攻击面就不可能安全地部署持久记忆。

**落地程度说明**：记忆安全目前几乎完全是学术研究领域。本文件讨论的攻击框架、防御机制和治理模型均来自论文和学术调查，没有在主流记忆框架中落地。以下内容的价值在于理解攻击面和思考方向，而非直接可用的工程方案。

## 内部结构

### 六阶段威胁模型

2026-04 发表的安全调查（arxiv 2604.16548）将记忆生命周期形式化为六个阶段，每个阶段都有独立的攻击面。威胁沿四个安全目标评估：完整性（integrity）、机密性（confidentiality）、可用性（availability）、治理性（governance）。

#### 写入阶段（Write）

攻击者通过各种途径将有毒信息注入 agent 记忆。这是研究最多的阶段，有三种主要攻击途径：

**对话注入**——攻击者通过正常对话向 agent 输入精心设计的内容，agent 的记忆系统将其作为正常交互提取并持久化。关键在于 agent 无法区分"用户提供的正常信息"和"攻击者植入的有毒指令"——它们在记忆系统看来没有区别。

**环境注入**——有毒内容不来自对话，而来自 agent 接触的外部环境：被污染的网页、恶意的工具返回值、被篡改的文档。Agent 从这些源中提取信息并存入记忆时，毒素被一并摄入。

**经验伪造**——MemoryGraft（2025-12）展示了一种更隐蔽的攻击：不直接注入有毒指令，而是**植入虚假的"成功经验"**。Agent 的程序记忆依赖历史上的成功案例来指导未来行为。如果攻击者成功伪造了几条"上次按方法 X 执行获得了好结果"的记录，agent 将在未来类似场景中倾向于使用方法 X——即使方法 X 实际上是恶意的。这种攻击利用的是 agent 的**语义模仿启发式**——模仿过去成功的做法。

**核心盲点**：调查检查了所有已发表的记忆架构，**没有任何一个实现了写入门控验证（write-gate validation）**。信息写入记忆之前没有内容审查、来源验证或信任评估。这意味着攻击者只要能让信息进入 agent 的感知范围，就很可能将其持久化为记忆。

#### 存储阶段（Store）

记忆存储后面临的问题不只是保持数据不被篡改，还包括压缩操作中的信息劣化。

**压缩放大**：当记忆系统对历史进行摘要压缩时，有毒内容不但不会被过滤，反而可能在摘要中获得更高的权重——因为它通常被精心设计为"看起来重要"的信息。多次压缩后，毒素占比可能反而上升。

**静默过时**：非对抗性的存储问题。高置信度的事实随时间变得错误（用户换了工作、项目改了技术栈），但系统不知道，继续以高置信度提供过时信息。这不是安全攻击，但效果类似——agent 基于错误记忆做出错误决策。

**跨用户污染**：在多用户共享存储的系统中，一个用户的记忆可能被错误地检索并应用到另一个用户的会话中。这通常是隔离不足导致的工程缺陷，而非主动攻击。

**缺失的防御**：版本化快照和回滚能力——允许在发现污染后恢复到已知良好的记忆状态。调查发现几乎没有系统实现了真正的版本化回滚。

#### 检索阶段（Retrieve）

即使存储的记忆本身没问题，检索过程也可以被操纵。

**RAG 投毒**：污染检索数据源，使得有毒内容在语义搜索中排名靠前。GRAGPoison 针对图记忆展示了一种结构性攻击：利用知识图谱中的共享关系，一次攻击可以影响多个查询路径，**攻击成功率高达 98%**。图结构增强检索精确性的同时，也放大了结构性污染的影响范围——这是图记忆特有的安全悖论。

**控制流劫持**：检索到的有毒记忆覆盖用户的显式指令，改变 agent 的工具选择和执行顺序。这发生在检索内容和当前指令存在语义冲突时——如果系统没有明确的优先级规则（当前指令 > 检索记忆），agent 可能听从被污染的记忆而非用户。

#### 执行阶段（Execute）

检索到的记忆影响 agent 实际行为的阶段。

**关键风险**是信息流控制的缺失——从外部检索到的记忆内容和系统指令在 context window 中混在一起，模型无法区分两者的权限级别。理论上，检索到的记忆不应该有权覆盖系统指令，但在实际的 context window 中，两者只是不同位置的文本。

#### 共享阶段（Share）

在多 agent 系统中，记忆通过共享存储或 agent 间消息传播。

**蠕虫式传播（理论攻击路径，尚无实际案例）**：一个被污染的 agent 将有毒记忆写入共享存储，其他 agent 检索并内化这些记忆，然后各自进一步传播。学术调查指出这一路径在多 agent 共享存储的架构中理论上可行，且内部通道（agent 间通信）比用户可见输出更难审查。但截至 2026-04，没有公开的实际攻击案例。

**隐私泄露**：在共享记忆系统中，Agent A 在服务用户 A 时积累的私密信息，可能通过共享存储被 Agent B 检索并暴露给用户 B。

#### 遗忘/回滚阶段（Forget/Rollback）

**删除验证的缺失**：调查发现没有任何已检查的架构实现了删除后验证——当用户要求删除某条记忆时，系统没有机制确认记忆已被从所有存储基底（向量库、图数据库、缓存、备份）中彻底清除。

**级联遗忘的困难**：删除一条事实后，基于该事实推导出的结论也应该失效。但在非结构化的向量存储中，追踪信息间的推导关系几乎不可能。图记忆的关系结构在这方面有理论优势，但也未见实际实现。

**合规压力**：GDPR 的"被遗忘权"要求用户能要求删除与其相关的所有数据。但在 agent 记忆系统中，用户数据可能经过多轮摘要和抽象，散布在多个存储层中，完全删除在技术上极具挑战。

### 记忆主权（Mnemonic Sovereignty）

调查提出"记忆主权"作为统一的治理框架——系统必须对以下五个维度有可验证的、可恢复的治理能力：

1. **谁可以写入**（write authorization）
2. **谁可以读取**（read authorization）
3. **何时允许更新**（update policies）
4. **哪些状态必须可审计**（audit requirements）
5. **哪些状态可以被遗忘**（forgetting policies）

这五个维度跨越四个经典安全目标：

| 安全目标 | 定义 | 当前研究覆盖度 |
|---------|------|--------------|
| **完整性** | 防止投毒、偏差和腐蚀 | 中（写入和检索阶段研究较多） |
| **机密性** | 防止未授权的提取或跨边界泄露 | 中 |
| **可用性** | 在正确上下文中检索关键记忆 | 极低（~5%，拒绝检索攻击仍是理论性的） |
| **治理性** | 可审计的来源、更新历史和回滚能力 | 极低（~5%） |

**关键发现**：没有任何已发表的架构实现了调查识别出的全部九个治理原语。研究集中在写入和检索阶段，存储和遗忘阶段的覆盖率加起来不到 5%。可用性攻击（denial-of-retrieval、延迟投毒）和良性持久失败（非对抗性的记忆腐蚀）几乎没有被研究。

### 防御现状

#### 已提出的防御机制

**写入端**：内容写入记忆前进行验证——检查来源可信度、内容合规性。为每条记忆附加显式的来源标签（provenance tagging），区分"用户直接输入"和"从环境中提取"。

**存储端**：版本化快照，允许回滚到已知良好状态。压缩审计——在摘要过程中保留信息血统（lineage），追踪每条摘要的原始来源。保留策略分层——区分"必须可审计"（如医疗记录）和"用户可删除"（如偏好设置）的记忆。

**检索端**：可认证的鲁棒性保证（certifiable robustness）。基于聚类的信任感知检索路由。基于激活的异常检测。跨多个记忆源的共识验证。

**执行端**：信息流控制（information-flow control）——防止检索到的外部内容覆盖系统的显式指令。在技术实现上，这意味着 context window 中不同来源的内容需要有不同的权限级别——目前没有 LLM 原生支持这种机制。

**A-MemGuard**（2026）：首个针对 agent 记忆的主动防御框架。它解决的一个特殊困难是——注入的有毒记忆在被单独审计时看起来完全正常，只有在特定上下文中被检索到时才激活恶意效果。这使得基于内容审查的防御（在写入端检查每条记忆）不够有效，需要在检索和执行阶段增加上下文感知的检测。

#### 防御的根本困境

2025 年的学术评估（Nasr et al.）在实验室条件下测试了 12 种 prompt injection 防御机制，自适应对手的攻击成功率超过 90%。需要注意这是受控实验环境下的数据——实际生产环境中攻击者面对的约束（访问频率限制、行为监控等）会降低实际风险。但这个数字仍然说明，纯基于规则或检测的防御思路存在根本局限。

调查指出了一个成熟度缺口：记忆安全的防御还没有引入 LLM 自身作为安全工具——用 LLM 进行红队测试、自动化审计、遗忘验证。在传统安全领域，攻防是对称的（攻击者和防御者使用类似的工具）；在记忆安全领域，攻击者已经在使用 LLM 设计攻击，但防御者还主要依赖规则和启发式。

### 架构选择的安全含义

不同的记忆架构（topics/memory-architecture.md）带来不同的安全特征：

**向量存储**：攻击面相对简单（主要是语义相似度操纵），但防御也困难（非结构化，难以追踪信息血统）。

**图存储**：增强了检索精确性，但引入了关系级攻击面——学术研究（GRAGPoison）在实验室条件下展示了利用图共享关系结构的攻击路径，一次攻击可影响多个查询。图结构理论上有利于级联遗忘（可以追踪推导关系），但实际没有系统利用这个优势。

**Agent 自主管理（Letta 式）**：引入了一个独特风险——如果 agent 的记忆管理能力被操纵（通过污染 agent 的记忆管理规则），攻击者可以间接控制 agent 的整个记忆策略。另一方面，agent 自主管理也意味着 agent 有可能自我检测记忆异常——这在被动管理的系统中不可能。

**元数据承载架构**（如 Graphiti 的双时间线模型）在安全治理上有结构性优势——每条记忆自带来源、时间和有效性元数据，使审计和回滚成为可能。但"必要不充分"——有元数据不等于有治理。

## 当前状态（截至 2026-04）

**学术**：研究集中在攻击端（写入和检索阶段），防御端严重不足。A-MemGuard 是首个系统性防御框架。ICLR 2026 的 MemAgents workshop 将记忆安全作为核心议题之一。

**产品**：没有主流记忆框架将安全作为核心设计原则。Mem0、Letta、Zep 的文档和功能中，安全主要是"未来路线图"项目。OpenMemory MCP 的本地优先（local-first）模式是一种回避策略——通过将记忆保留在本地来减少攻击面，但不解决本地记忆本身被污染的问题。

**企业**：LinkedIn CMA 等企业系统的安全细节未公开，但可以推测企业内部有更多治理机制。企业场景的核心需求是可审计性和合规性，这和学术界关注的攻击-防御对抗是不同的侧重点。

**差距**：记忆安全的成熟度远落后于模型安全（guardrails、alignment）和应用安全（prompt injection 防御）。这可能成为 agent 记忆从实验走向企业生产的最大阻碍。

## 关键权衡

**1. 安全 vs 可用性**

越严格的写入门控意味着越多的合法信息被拒绝写入（false positive）。在医疗场景中，一条被误拒的关键过敏信息可能造成比记忆污染更严重的后果。

**2. 审计深度 vs 成本**

完整的信息血统追踪（每条记忆的来源链、压缩历史、被谁检索过）提供最强的治理能力，但存储和计算成本随审计深度线性增长。

**3. 遗忘彻底性 vs 系统复杂性**

真正的"级联遗忘"（删除一个事实后所有推导结论也失效）需要追踪信息间的因果关系。在非结构化存储中这几乎不可能；在图存储中理论上可行但极大增加系统复杂性。

**4. 中心化存储 vs 本地优先**

中心化存储便于管理和审计，但攻击一个中心存储可以影响所有用户。本地优先（OpenMemory MCP 模式）减少了单点攻击面，但让全局的安全策略执行变得困难。
