## 规则（先读完再开始）

1. **不查任何资料**。包括不打开 dimensions/layers 已有文件、不搜索、不问 Claude。这是 baseline 快照，不是答题。
2. **写不出来就留空**。空白本身是有信息量的——它精确指出了你的盲区位置。不要为了不空着而硬凑词。
3. **写完整句子，不要堆名词**。"我知道 ReAct" 不算骨架。"ReAct 是 reasoning 和 acting 交替的循环模式，每轮模型先输出 thought，再选 action，观察结果后进入下一轮，相比纯 chain-of-thought 多了和外部环境交互的能力" 才算骨架。
4. **顺序不限，可以分多次完成**。但不要跨越太久——拖一周，前面写的就不是 baseline 了。
5. **写完一节立刻自评等级**，不要等全部写完再回头评。
6. **与之后阶段 1 的关系**：这份文档是"before 快照"，永久保留不再修改。阶段 1 你会把骨架的精华迁移并扩充到正式的 dimensions/layers 文件里。

## 第一部分：四个工程层级

### L1 Agent 内核

> 提示问题：这一层在解决什么问题？有哪些主流的 agent 循环模式？context 工程有哪些关键决策点？

**凭记忆写出**：

这一层聚焦在Agent本体的运行上，Agent循环模式：
- 从最开始的ReAct，思考 - 行动 - 观察 的循环模式
- 然后由于模型内化了Reasoning，就变成了思考 - 行动的循环，即思考清楚再行动
- 还有，Plan and Excute 模式，但是这种模式在长链路任务上很容易出现误差累加

prompt工程在于思考如何写出更优质的提示词，而context工程在于如何在Agent运行过程中，在合适的时机给到合适的内容。
context的存在，原因在于超长的提示词会分散大模型的注意力，且冗余的内容会给大模型的决策带来污染。

一些关键的处理方式有：
1. 设计锚点进行摘要，以压缩历史上下文
2. 上下文中选择一些合适的信息写出。例如，工具响应的结果或者历史的对话理论上也可以，但是这种写出需要可逆，即我可以在需要的时候检索回来
- 这里关于写出的原则，我设置的是超过6轮以上的工具会进行写出
- 但不是所有工具结果都会进行写出，我选择数据型的工具进行写出
- 一些重要决策类工具都会保留结果
3. 相对独立的场景封装成subagent，单独去执行，避免污染主agent的上下文
4. 还有本身skill的设计，就是所谓渐进式披露，也是一种context工程
5. 还有RAG的应用，外挂知识库也是context工程的一部分。
- RAG的发展也经历了：naive 向量召回，误差率较高已经弃用了
- advanced rag : 输入的扩写后，embedding召回 + 重排序
- agentic rag ：由大模型agent 进行信息的召回

### L2 与世界的接口

> 提示问题：tool use 的关键设计决策有哪些？MCP 解决了什么问题？RAG 的主流架构和常见失败模式？memory 系统的几种主流方案？

**凭记忆写出**：

世界的接口，也就是agent链接世界的方式。
经历过的和正在经历的
api接口schema适配，在LLM初期需要agent主动去适配各种api接口，100个agent需要100套适配。知道后来出现了mcp协议，将mcp注册到注册中心，只披露简介，由client端去发现解决问题所需要的mcp，各个agent只需要接入统一的client端即可。 client相当于一个工具集的代理。简化了工具的治理。

再后来出现了CLI协议，不过cli的出现并不是替代mcp协议，两者一定程度上是互补的关系，但在我认知中他们不是一个层面的概念。
cli是一套agent所熟悉的感知计算机环境的方式，环境也包含了工具（ls/grep/cat/python/bash等等）
如果需要类比的话，如果把mcp比做搜索引擎的输入框的话，那么cli就是鼠标和键盘。
我不太确定我的比喻和理解是否恰当。

但是除了两者的表象之外，mcp对于agent来说像瑞士军刀，它是一个工具套件，对于依赖mcp多个工具去解决问题，且工具使用之间存在依赖关系的话，mcp是比cli更有场景。
因为cli偏向于一次性的执行获得结果，而mcp对于agent的多个步骤执行过程信息是在同一个上下文中的。

cli对于token的节省是可观的，可以借助计算机工具去检索相关的相对精准的信息，信息是可以以索引形式提供给关联文件，这样agent可以在需要的时候选择性获取文档。
也就是所谓的渐进式披露。

RAG 的主流架构和常见失败模式？
- naive rag
- advanced rag
- modular rag
- agentic rag

naive rag 是初代rag，错误率较高已经被淘汰

advenced rag 在输入、匹配、召回进行了优化
输入层扩写或者重写，针对问题模型生成答案去匹配
然后便是粗排，bm25
精排 rerank模型去对粗排结果进行更精准排序

modular rag
主要是进行了模块化

agentic rag
字面意思，由agent来决策何时获取信息，如何获取信息。

rag的细节还有待学习


memory 系统的几种主流方案？
要说memory分类的话，有很多分法，例如：
- 主动记忆：memory外写，模型根据需要去进行加载
- 被动记忆：上下文记忆

memory的压缩有很多手段，例如：
锚定式文档摘要：满足一定条件，例如token超过一定量，对话轮数超过一定阈值，对上文按照设计好的要点进行总结，作为背景
工具结果压缩：数据型工具结果可以外写到数据库，在需要的时候使用工具检索。
- 当然，此处工具压缩，不是无脑压缩，它是有原则。大致是，决策与目标不压缩，而执行过程和原始数据压缩。
- 这有点像产品经理，我需要产品向正确方向发展，那么对我来说正确的决策和目标是比执行过程更重要的。

subagent：对于相对独立的业务模块，主agent只关系结果的业务部分，抽象出subagent，避免其记忆污染主agent
